# Atoms Demo 实现说明

## 项目目标

本项目的目标是实现一个参考 Atoms 工作流的 AI App Builder Demo。用户可以在左侧输入应用需求，通过 Engineer、Team、Race 三种模式生成 React 应用，并在右侧应用查看器中预览、编辑、修复、发布和分享结果。

Demo 的重点不是做一个完整商业化平台，而是尽可能还原 Atoms 文档中强调的核心体验：自然语言生成应用、多智能体协作过程、实时预览、错误修复、版本留存和可分享交付。

## 实现思路

### 1. 工作台优先，而不是营销首页

首屏直接进入工作台，左侧承载项目输入、模式选择、对话流和项目历史，右侧承载应用预览与生成结果。这样更符合 Atoms 类产品的使用场景：用户进入后希望马上表达需求并看到可运行结果，而不是先浏览介绍页。

### 2. 以对话流串联生成过程

用户每次 Build 都会把本轮输入记录进对话流，并在生成过程中实时追加智能体进度。这样可以让用户理解 AI 正在做什么，也保留“在当前项目基础上继续迭代”的上下文感。

目前对话流承担三类信息：

- 用户输入的需求。
- AI 生成过程中的阶段性进度。
- 生成完成后的总结、版本信息和后续可执行动作。

### 3. 真实 LLM 与本地兜底并存

服务端 API 默认支持 DeepSeek，也保留 OpenAI 配置路径。生成接口要求模型输出结构化 JSON，再经过 schema 校验和文件安全检查后进入 Sandpack 预览。

同时保留本地 fallback 生成器：当部署环境没有配置 API Key，或模型调用失败时，仍可以生成一个可运行的 React 原型。这个取舍保证了评审、演示和部署预览不会因为密钥问题完全不可用。

### 4. 生成文件进入受限沙箱

生成结果只允许落在 Sandpack 可运行的 React/Vite 文件结构内，核心文件包括：

- `/src/App.tsx`
- `/src/main.tsx`
- `/src/styles.css`
- `/package.json`

依赖、路径和外部脚本都做了约束，避免模型生成任意不可控文件或引入危险资源。

### 5. 本地持久化优先

项目数据、版本、运行记录和发布快照保存在 IndexedDB。选择 IndexedDB 的原因是它不需要额外后端或数据库，适合快速交付一个可体验的 Demo。

这也意味着当前数据主要保存在用户自己的浏览器里，暂不支持跨设备同步和多人协作。

## 关键取舍

### 使用 IndexedDB，而不是 Supabase

优点：

- 不需要维护数据库 schema、权限和登录态。
- 本地即可完整体验项目历史、版本保存和发布快照。
- 更适合笔试 Demo 的交付节奏。

代价：

- 不支持多设备同步。
- 公开分享只能通过压缩快照链接实现，不是真正的云端公开项目。
- 后续如果要做生产级产品，需要迁移到云端存储。

### 使用 Sandpack，而不是自己实现代码运行环境

优点：

- React 代码可以直接在浏览器中预览。
- 内置编辑器、文件系统和依赖解析能力。
- 降低手写 iframe 构建系统的复杂度。

代价：

- 对可运行项目类型有边界，更适合前端单页原型。
- 后端 API、数据库连接、复杂构建流程需要额外扩展。

### 使用结构化输出，而不是直接展示模型文本

优点：

- 可以稳定提取项目名、文件列表、智能体事件、检查结果。
- 更容易做 schema 校验和自动修复。
- UI 可以根据结构化字段展示进度、代码、预览和版本。

代价：

- prompt 和解析逻辑更复杂。
- 模型返回格式不稳定时需要 JSON 修复与兜底策略。

### 先做单用户体验，而不是完整协作系统

当前 Demo 更重视“用户输入需求到生成可预览应用”的闭环，所以没有实现团队成员权限、评论、实时协同编辑和云端项目空间。

这些能力很重要，但属于第二阶段扩展。

## 当前完成程度

### 已完成

- Engineer、Team、Race 三种模式切换。
- 左侧对话流记录用户输入、AI 进度和生成结果。
- Build 后自动清空当前输入框，避免重复提交同一需求。
- Team 模式下生成多个智能体的任务进度。
- Race 模式下生成候选结果，并允许选择候选进入主版本。
- `/api/generate` 生成应用文件。
- `/api/polish` 优化用户 prompt。
- `/api/repair` 根据问题描述修复当前文件。
- DeepSeek 配置链路。
- OpenAI 可选配置链路。
- 本地 fallback 生成器。
- Sandpack App Viewer 预览和代码编辑。
- 桌面端、移动端预览切换。
- Visual Tokens 快速调整生成应用视觉风格。
- Issue Report 输入报错或修改方向并触发修复。
- 项目列表和版本历史持久化。
- 发布快照链接。
- 分享权限标签。
- 导出 zip。
- 简单登录/注册入口，支持本地账户体验。
- 单元测试覆盖 schema、文件安全、发布快照、IndexedDB 迁移和 LLM mock 返回。
- README 中包含本地运行、部署和环境变量说明。

### 部分完成

- 登录/注册目前是本地 Demo 级能力，不是接入真实云端账号系统。
- 发布功能是压缩快照链接，不是真正上传到云端并生成公开项目页。
- Race Mode 有候选生成和选择，但还没有自动评分、对比维度解释和模型推荐。
- Issue Report 可以触发修复，但还没有做到完整的运行时错误自动捕获和一键定位源码行。
- Visual Tokens 可快速调整主视觉，但还不是完整设计系统编辑器。

### 暂未完成

- 多人协作和权限管理。
- 云端项目同步。
- 真实自定义域名发布。
- 生产级用户认证。
- 生产级计费、额度和使用统计。
- App World 或公开作品广场。
- 后端服务、数据库、第三方 API Connector 的真实集成。
- Playwright E2E 自动化验收。
- 更完整的安全沙箱审计。

## 如果继续投入时间，如何扩展

### P0：提升生成稳定性

优先级最高。AI App Builder 的核心价值在于生成结果稳定可运行。

计划：

- 增强结构化输出 schema。
- 增加模型返回自动修复重试。
- 对生成文件做更严格的路径、依赖和语法检查。
- 在服务端增加更清晰的错误分类：模型错误、网络错误、JSON 解析错误、文件安全错误、运行时错误。
- 在前端把错误反馈给用户，并提供一键修复入口。

理由：

如果生成链路不稳定，后续的协作、发布和作品广场都没有足够可靠的基础。

### P1：云端项目与真实账号

第二优先级是把本地 Demo 变成可跨设备使用的产品雏形。

计划：

- 接入 Supabase 或类似后端。
- 将项目、版本、发布记录从 IndexedDB 迁移到云端。
- 保留 IndexedDB 作为离线缓存。
- 增加真实登录、注册、登出和会话管理。
- 支持用户自己的项目空间。

理由：

只有项目进入云端，分享、发布、历史版本和多人协作才有生产级基础。

### P2：发布系统升级

当前发布是快照链接，适合 Demo，但不适合真实产品。

计划：

- 增加公开项目页。
- 为每个发布版本生成稳定 URL。
- 支持回滚到历史发布版本。
- 支持公开、私有、仅链接可见等权限。
- 后续可接入自定义域名。

理由：

Atoms 类产品的最终价值是“把生成结果交付给别人使用”，所以发布体验会直接影响产品完整度。

### P3：更完整的 Agent Team 和 Race Mode

计划：

- 给不同智能体更明确的职责和输出结构。
- Team 模式增加任务拆解、设计评审、代码实现、QA 检查等阶段。
- Race Mode 增加候选对比面板。
- 自动给候选打分，例如可运行性、视觉完成度、需求覆盖度、代码简洁度。
- 支持用户将多个候选的优点合并。

理由：

这部分最能体现 Atoms 文档里的差异化体验，但需要建立在稳定生成和预览基础上。

### P4：设计系统与编辑体验

计划：

- 扩展 Visual Tokens，支持字体、圆角、间距、阴影、组件密度。
- 增加页面结构树。
- 增加选中元素后局部修改。
- 支持“只修改按钮样式”“只重写首屏文案”等局部编辑。

理由：

用户生成后通常不会一次满意，低成本精修是提升可用性的关键。

### P5：测试与质量保障

计划：

- 增加 Playwright E2E。
- 覆盖从输入 prompt、生成、预览、切换移动端、修复、发布、刷新恢复历史的完整链路。
- 对常见生成失败样例建立回归用例。
- 部署前跑 `pnpm exec tsc --noEmit`、`pnpm lint`、`pnpm test`、`pnpm build`。

理由：

AI 生成链路不确定性高，持续回归测试可以避免修一个体验时破坏另一个核心流程。

## 总结

当前版本已经完成了一个可部署、可演示、可迭代的 Atoms 风格 AI App Builder Demo。它覆盖了从自然语言输入、智能体进度、应用生成、实时预览、错误修复、版本保存到发布分享的主要闭环。

主要取舍是优先完成单用户 Demo 闭环，用 IndexedDB 和 Sandpack 降低后端复杂度，用 fallback 生成器保障无 Key 场景下仍能体验完整流程。后续如果继续投入，最应该先加强生成稳定性和云端项目系统，再扩展发布、协作、Race 评分和更细粒度的可视化编辑能力。
